01 · Cursor 4 / 20

The AI-native
code editor

Cursor brings an LLM directly into the edit loop — not as a sidebar chatbot, but as a first-class collaborator that reads your entire codebase and edits files alongside you.

  • ⌘K / Ctrl+K — inline edit: describe what you want, Cursor diffs the result
  • ⌘L / Ctrl+L — multi-turn chat with full codebase context
  • Agent Mode — give it a task; it plans, edits files, runs terminals, iterates
  • Tab Completion — multi-line completions that respect your code style
  • .cursorrules — project-level instructions applied to every AI interaction
  • macOS · Windows · Linux — native installers for all three platforms; same feature set everywhere
// ⌘K → "make this async + add error handling"

async function fetchUserData(userId) {
  try {
    const res = await fetch(
      `/api/users/${userId}`
    );
    if (!res.ok) {
      throw new Error(`HTTP ${res.status}`);
    }
    return await res.json();
  } catch (err) {
    console.error('fetchUser failed', err);
    throw err;
  }
}

01 · Cursor — Under the Hood 5 / 20

How Cursor works technically

VS Code Extension Support

Cursor is a fork of VS Code — it runs the full VS Code extension API. The vast majority of existing extensions install and work without modification.

  • Import your existing VS Code profile and extensions in one click
  • Language servers (LSP) run natively — TypeScript, Rust Analyzer, Pylance all work
  • Debugger adapters, linters, formatters, themes — fully compatible
  • Extensions using terminals or the file system work as expected
  • Only exception: extensions that hook into VS Code's AI/Copilot API directly

Codebase Indexing

When you open a repo, Cursor reads all your code once and builds a searchable map — so the AI always knows where things are.

Like a book's index — instead of scanning every page each time, you jump straight to what's relevant.

Chunking Breaks code into meaningful pieces (functions, classes) instead of arbitrary blocks
Embeddings Each piece gets a unique fingerprint stored locally — your code never leaves your machine
Retrieval When you ask a question, the most relevant pieces are pulled in automatically

Tokenization & Context Window

AI models have a limit on how much they can "see" at once. Cursor decides what fits — always keeping the most useful code in view.

Like a desk with limited space — you keep today's work front and center, and file away the rest within reach.

Tokens The unit the AI reads in; roughly 3–4 characters each. Every request has a cap on how many fit
Context budget Your current file is always included; the rest of the space is filled with the most relevant code from the index
Bigger models Claude's 200k+ token window means Cursor can fit entire files instead of just snippets

01 · Cursor — Context 6 / 20

Making the most of
the context window

The AI can only see what fits in its context. Too much clutter and it loses focus — too little and it lacks the information to help you well.

  • Keep rules & CLAUDE.md concise — they load on every single turn and eat into the budget before your question is even asked
  • Attach only what's needed with @ — referencing the whole repo costs far more than referencing the one file that matters
  • Start a fresh session for unrelated tasks — prior conversation history is expensive context that the model must re-read each turn
  • Use subagents for large tasks — each spawned agent gets its own isolated context window, keeping the main session clean
  • Run /compact when sessions grow long — summarises history in place so you keep the key facts without the full transcript

02 · Skills 11 / 20

Graphify — Your Codebase as a Knowledge Graph

Maps your entire project into a queryable graph so AI skips the file-reading marathon and goes straight to what matters. Built locally — no API calls during extraction.

Like Google Maps for your repo — you query the map, not every street.

Why It Exists

  • →Reading full file trees for every question wastes tokens and is slow
  • →Graphify builds the graph once; queries answer with 50%+ fewer tokens
  • →31 languages, Salesforce metadata, PDFs, Office docs
  • →AST extraction runs locally — no code leaves your machine

Key Commands

graphify .

Build or rebuild the graph for the current directory

graphify query "how does X work?"

Ask a natural-language question about the codebase

graphify path "ClassA" "ClassB"

Trace the shortest connection between two nodes

graphify . --update

Re-extract only files changed since last run

What It Produces

  • graph.json — machine-readable graph; what AI queries against
  • GRAPH_REPORT.md — architecture overview: god nodes, surprising connections, design rationale
  • graph.html — interactive browser visualization with search & filtering
  • graphify-out/wiki/ — full markdown wiki (--wiki flag)

Add graphify-out/ to .gitignore — generated output, never commit.

Why MCP matters

Before MCP, every AI-to-tool connection needed its own custom code. MCP is the universal plug — one standard that lets any AI work with any tool, out of the box.

1,000+
Pre-built connectors ready to install — Slack, GitHub, Salesforce, databases, and more
0 custom
code
One standard replaces every one-off integration you'd have had to build by hand
Any AI,
any tool
Claude, GPT, Cursor, VS Code — they all speak MCP, so your connectors work everywhere
⚠️

Don't go overboard. Every active MCP adds tools to the model's context. Too many and the AI starts struggling to pick the right one — or gets confused entirely. Enable only what you actually need for the task at hand.

04 · Subagents 15 / 20

Parallel work,
single result

Subagents are specialized child agents spawned to work on discrete sub-tasks in parallel — letting complex workflows scale beyond what a single context window can hold.

  • Each subagent gets a focused, self-contained prompt
  • Parallel execution cuts wall-clock time dramatically
  • Subagents can use different tools or models than the orchestrator
  • Context isolation prevents earlier conversation from polluting results
  • Adversarial verify — multiple agents review the same finding independently
// Orchestrator: spawn parallel subagents

const results = await parallel([
  () => agent(
    "Review auth module for bugs",
    { phase: 'Review' }
  ),
  () => agent(
    "Review payments module for bugs",
    { phase: 'Review' }
  ),
  () => agent(
    "Review API layer for bugs",
    { phase: 'Review' }
  ),
]);

const confirmed = results
  .filter(Boolean)
  .flatMap(r => r.findings);
